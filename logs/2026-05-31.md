## Decisiones técnicas

| Decisión | Razón |
|---|---|
| Python 3.14 (no 3.12) | Usuario ya tiene 3.14 instalado, adaptar dependencias es más rápido |
| `>=` en lugar de `==` en requirements | Versiones pinneadas viejas no tienen wheels para Python 3.14 |
| `supabase` SDK removido de requirements | No se importa en ningún lado; nueva versión arrastra pyiceberg (necesita MSVC en Windows) |
| `google-cloud-aiplatform` comentado | Mismo problema pyiceberg — se reactiva en Week 4 |
| schema.sql + rls_policies.sql fusionados | Supabase advertía sobre tablas sin RLS al aplicar en archivos separados |
| `workers.pin` → `pin_hash` | Regla de seguridad: PINs se hashean con bcrypt, nunca en texto plano |
| Connection pooler puerto 6543 + ssl=require | Puerto 5432 da TimeoutError desde redes locales; Supabase recomienda pooler para entornos no-persistentes |
| GitHub Actions con JSON key (no Workload Identity) | Más simple para hackathon; Workload Identity requiere configuración extra en GCP |

---

## Problemas encontrados y soluciones

| Problema | Causa | Solución |
|---|---|---|
| `pydantic-core` no compila | Python 3.14, no hay wheels para versión pinneada | Cambiar `==` por `>=` en requirements |
| `pyiceberg` no compila | Dep transitiva de `supabase` SDK y `google-cloud-aiplatform` | Remover supabase SDK, comentar aiplatform |
| `TimeoutError` en `/health/db` | Puerto 5432 bloqueado + falta SSL | Usar connection pooler (6543) + `ssl=require` |
| `.env.example` bloqueado | Restricción de permisos en archivos dot | Usuario crea manualmente con instrucciones |
